### 错误总结

**核心现象**：MySQL 8.0.46 容器初始化时，创建自举线程失败，导致数据字典初始化中断，服务器中止退出，容器反复重启均无法恢复。

---

### 错误根源

**操作系统层拒绝**：Linux 内核返回 `errno: 1`（`EPERM`，Operation not permitted），主动禁止 MySQL 进程执行 `clone()` 系统调用来创建线程。

**触发条件**：MySQL 8.0 在首次启动执行 `--initialize` 时，必须创建一个内部线程来完成数据字典的引导构建（bootstrap），该线程创建请求被内核安全机制拦截。

**环境关联**：
- 发生在 Docker 容器内部，宿主为 macOS（Docker Desktop）。
- 报错镜像基于 Oracle Linux 9（`el9`），该基础镜像默认的内核安全策略（seccomp 或 cgroup pids 限制）比旧版更严格。
- MySQL 8.0.46 和 8.4.0 等较新版本的 Docker 官方镜像均有此问题报告，而基于 Oracle Linux 8 的旧镜像不会触发。

### 详细解释：MySQL 初始化报错 `Can't create thread to handle bootstrap (errno: 1)`

---

#### 一、错误码 `errno: 1` 的精确含义

Linux 系统错误码 `1` 对应宏 `EPERM`，全称 **“Operation not permitted”**。  
**这不是“资源不足”（资源不足会是 `ENOMEM` 错误码12或 `EAGAIN` 错误码11）**，而是内核安全机制主动拒绝操作——即便进程有 root 身份，内核仍然判定该操作不被允许。

---

#### 二、MySQL 此时到底在做什么

MySQL 8.0 首次启动时，需要执行 **初始化流程**：
1. 进程从 `--initialize` 参数开始自举。
2. 创建一个内部线程去执行 **数据字典的 bootstrap 构建**（即在 `mysql` 表空间中创建系统表）。
3. 构建完成后将元数据持久化到数据目录 `/var/lib/mysql/`。

**而在这个流程中，创建线程这个最基础的 OS 操作直接失败了**，后续所有步骤自然全部崩溃。

---

#### 三、关键问题定位：为什么 MySQL 无法创建线程？

`EPERM` 的来源只有两种可能：**安全策略拦截** 或 **cgroup 资源控制器拦截**（后者在特定条件下也会返回 EPERM）。

| 限制类型 | 说明 | 与当前环境的关联 |
| :--- | :--- | :--- |
| **seccomp 过滤** | Docker 默认使用 seccomp 安全配置文件，限制容器内可调用的系统调用。默认 profile 允许 `clone`，但如果被修改或使用旧版 Docker Desktop（macOS 上依赖 LinuxKit 虚拟机），可能会误伤 MySQL 的 `clone` 调用。 | 非常高。Docker Desktop for Mac 在特定版本曾有过 seccomp bug。 |
| **AppArmor / SELinux** | 宿主机强制访问控制系统拒绝权限。 | 中等，但 macOS 上的 Linux 虚拟机通常未启用复杂 MAC，除非自定义。 |
| **cgroup pids 限制** | Docker 可通过 `--pids-limit` 限制容器内总进程/线程数。若该值设为极低（如 10），MySQL 初始化需创建的线程可能立刻触及上限，返回 `EPERM`。 | 可能，取决于 compose 或 run 命令是否设置了该参数。 |
| **Linux 内核 `clone` 权限检查** | 特定内核版本在某些命名空间配置下，非特权进程执行 `clone` 可能因 capability 不足被拒。 | 中高，Docker 容器的默认 capability 集合已包含 `CAP_SYS_ADMIN` 等，一般不会缺，但若被手动删减或者内核参数异常则可能触发。 |

**以你的错误日志上下文判断，关键问题 99% 落在第一种情况——容器运行时的安全策略阻止了 `clone` 系统调用，导致 MySQL 初始化线程无法产生。**

---

#### 四、为什么重启、清数据目录都没用？

因为错误发生在 **操作系统内核层面**，与数据目录状态无关。你之前的尝试：
- 清空 volume 数据目录 → 只是让 MySQL 能进入初始化流程，但初始化依然会立即卡在创建线程上。
- 重启容器 → 重试完全相同的系统调用，依然被同一个内核策略拒绝。

所以错误会 **完全复现**，且进程号变化（80 变 79）只说明这是又一次全新的失败尝试。

---

#### 五、问题的根源层

最终追究，限制的施加者并非容器内的 Linux，也不是 macOS 本身，而是 **Docker Desktop 为提供容器隔离而运行的 Linux 虚拟机内核**。  
**根本矛盾是**：MySQL 8.0 的初始化流程需要执行某个被当前容器的 seccomp/default 安全策略所阻止的系统调用。  
因此，要解决该问题，必须调整容器的安全上下文，让该调用放行。

---

#### 六、一句话总结关键问题

**关键问题不在 MySQL 配置，不在数据目录，不在宿主机 macOS 的文件权限，而在于 Docker 容器的内核级安全限制（极可能是 seccomp profile）直接拒绝了 MySQL 创建自举线程所需的系统调用。**
